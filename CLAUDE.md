# CLAUDE.md — 片付けアプリ「自分のご機嫌を、5分でとるアプリ」

このファイルは **Claude Code 向けのプロジェクト案内書**です。
このリポジトリで作業するときは、まずここを読んで、下の「設計上の約束」を守ってください。

---

## これは何のアプリ?
1日5分の小さな片付け習慣で「できた私」を増やし、**自己肯定感を高める**スマホ向け Web アプリ（PWA）。
- ターゲット：忙しい女性
- コンセプト：**片付けで人生（暮らし）が変わる**
- トーン＆デザイン：やさしい・あたたかい。配色は **クリーム × ピンク × セージグリーン**、丸ゴシック体。

## 技術スタックと方針（重要）
- **ビルド不要のプレーン構成**：HTML + CSS + バニラ JavaScript のみ。フレームワーク・Node・ビルドツールは**使わない**。
- データは原則 **ブラウザの localStorage**（端末内）に保存。
- **オンライン保存は Supabase（任意）**。`js/config.js` に設定されていれば有効、空なら端末内のみで普通に動く。
- **PWA**：`manifest.webmanifest` + `sw.js`（Service Worker）でインストール・オフライン対応。
- ⚠️ この方針を勝手に変えないこと。非エンジニアでも編集・無料ホストできることが最優先。
  - 新しいライブラリやビルド手順を**安易に増やさない**。必要なら必ず理由を説明してから。

## ファイル構成
```
index.html              アプリ本体（タブUIのコンテナ）
admin.html              管理者ページ（受講生の進捗一覧）
css/styles.css          見た目。配色は :root のCSS変数に集約
js/data.js              ★アプリ内の文章・コンテンツ（ここを編集すれば内容が変わる）
js/config.js            ★Supabaseの設定（URL・anonキー）。初期は空＝オンライン保存オフ
js/storage.js           localStorage の読み書き（記録・連続日数・ID・ニックネーム・why）
js/sync.js              Supabase との同期（受講生:書き込み / 管理者:閲覧）
js/app.js               画面描画・タブ切替・各機能のロジック（中心）
js/admin.js             管理者ページのロジック（ログイン＋一覧）
manifest.webmanifest    PWA設定
sw.js                   Service Worker（キャッシュ。更新時はCACHEの版数を上げる）
icons/                  アプリアイコン
README.md               人間向けの導入・セットアップ手順
EXERCISES.md            受講生向けの練習課題集（このアプリを育てるお題）
```

## このプロジェクトの使われ方（重要）
このアプリは講座の教材で、**受講生が Claude Code で「作る練習」をするための動く土台**です。
- 受講生から機能追加を頼まれたら、`EXERCISES.md` の課題に沿って**いっしょに実装を手伝う**スタンス。
- いきなり完璧に全部作らず、**小さく作って→ブラウザで確認→次へ**を促す。
- 上の「設計上の約束」（やさしいトーン・data.js集約・ビルドを増やさない）を保つこと。

## 画面（タブ）
`ホーム / おへや / きろく / しんだん / せってい`（`js/app.js` の `TABS` 配列で定義）
- ホーム：今日の5分タスク＋「できた!」ボタン＋なりたい自分バナー
- おへや：累計できた数でお部屋が育つ成長ビジュアル（Lv1〜6）＋「変化のあゆみ」
- きろく：連続日数・できた合計・カレンダー・バッジ
- しんだん：場所診断→チェックリスト
- せってい：名前/なりたい自分の変更・記録リセット・管理者ページへのリンク

## コンテンツの編集場所（`js/data.js`）
ほとんどの文章は `js/data.js` にまとまっている。ハードコードせず必ずここに足す。
- `tasks`：今日の5分タスク集
- `messages` / `doneCheers`：応援メッセージ
- `places` / `diagnosis` / `checklists`：場所診断とチェックリスト
- `badges`：連続日数バッジ
- `roomStages`：おへやの成長段階（`min`=その段階になる累計できた数、`name`/`word`/`change`）
- `whyQuestion` / `whyPresets`：「なりたい自分」設定の質問と選択肢
- 配色を変えるなら `css/styles.css` の `:root` 変数。

## localStorage のキー
`tk_done`（日付→完了）, `tk_checklist`, `tk_diagnosis`, `tk_meta`, `tk_uid`（端末ID）,
`tk_nick`（ニックネーム）, `tk_why`（なりたい自分）, `tk_room_lv`（演出用の最終レベル）
- キー名や形を変えるときは、既存ユーザーのデータが壊れないよう移行を考える。

## ローカルで動かす・確認する
`file://` で直接開くと Service Worker が動かないため、**簡易サーバ経由**で開く。
- 例：`python -m http.server 8000` → http://localhost:8000
- もしくは VS Code の「Live Server」拡張、`npx serve` など。
- 確認はブラウザの開発者ツールで**スマホ表示（幅375px前後）**にして行う。
- 主要動線：ニックネーム入力 → ホームで「できた!」→ おへやの成長 → しんだん → 管理者ページ。

## Supabase（管理者機能）
詳細は `README.md`「管理者機能のセットアップ」。
- テーブル `progress`。RLSで **匿名(anon)=登録/更新のみ、管理者(authenticated)=全件閲覧**。
- 管理者ログインは Supabase Auth（メール＋パスワード）。`admin.html` で使う。
- `js/sync.js` が進捗を upsert、`js/admin.js` が一覧表示。
- ⚠️ **anonキーは公開OK**（クライアントに置いてよい）。**service_role キーは絶対にクライアントやGitに置かない**。

## デプロイ
GitHub に push → **Vercel で Import**（静的サイトとして自動認識、ビルド設定は不要）。
- ファイルを更新したら `sw.js` の `CACHE`（例 `katazuke-app-v4`）の版数を上げると、利用者に反映されやすい。

## 設計上の約束（守ってほしいこと）
1. **やさしいトーンと配色**を保つ（クリーム×ピンク×グリーン／丸ゴシック）。
2. **文言・コンテンツは `js/data.js` に集約**（画面に直書きしない）。
3. **ビルドや重い依存を増やさない**（プレーンHTML/CSS/JSのまま）。
4. **localStorage のデータを壊さない**。
5. 変更したら**スマホ幅で表示確認**してから完了とする。
6. 秘密情報（service_role等）をコミットしない。
